Aim
Objective of this experiment

To implement Graph Traversal using Breadth First Search (BFS) in C, exploring all vertices of a graph in level-order fashion using a queue data structure — and to understand how BFS systematically visits all reachable nodes from a given source vertex, marking visited states and processing nodes layer by layer.

Learning Objective: Develop a clear understanding of how BFS explores graphs by visiting all nodes at the current level before moving to the next. Understand the role of the queue in maintaining exploration order, the visited array in preventing revisits, and why BFS achieves O(V+E) time complexity — compared to DFS's recursive depth-first nature.

By the end of this experiment, you will be able to trace all BFS iterations for any given graph, explain how nodes are enqueued and dequeued, identify the level-order traversal sequence, understand the adjacency matrix representation, and visualise queue operations — all animated step-by-step in the simulation.

Description
Theory & background

Breadth First Search (BFS) is a fundamental graph traversal algorithm that explores all vertices of a graph level by level — visiting all neighbours of a node before proceeding to their neighbours. It was formally described by Konrad Zuse in 1945 and later re-developed by Edward F. Moore in 1959 for finding shortest paths in mazes.

BFS uses a queue (FIFO) data structure to keep track of which vertex to explore next. When a vertex is visited, all its unvisited adjacent vertices are enqueued. This ensures the algorithm processes vertices in the exact order they were discovered — guaranteeing level-by-level exploration.

Key Insight: BFS is equivalent to exploring a graph as concentric rings radiating outward from the source. Level 0 is the source; Level 1 is its direct neighbours; Level 2 is their unvisited neighbours; and so on. This makes BFS ideal for finding shortest paths in unweighted graphs.

Program in C language:

bfs.cC
#include<stdio.h>

int queue[20], front = -1, rear = -1;
int visited[20];

void enqueue(int vertex)
{
    if(rear == 19) return;
    if(front == -1) front = 0;
    queue[++rear] = vertex;
}

int dequeue()
{
    if(front == -1 || front > rear)
        return -1;
    return queue[front++];
}

void BFS(int G[20][20], int n, int start)
{
    int i, vertex;
    for(i=0; i<n; i++) visited[i] = 0;

    enqueue(start);
    visited[start] = 1;

    while(front <= rear)
    {
        vertex = dequeue();
        printf("%d ", vertex);

        for(i=0; i<n; i++)
        {
            if(G[vertex][i]==1 && !visited[i])
            {
                enqueue(i);
                visited[i] = 1;
            }
        }
    }
}

int main()
{
    int G[20][20], n, i, j;
    printf("Enter number of vertices: ");
    scanf("%d", &n);
    printf("Enter adjacency matrix:\n");
    for(i=0; i<n; i++)
        for(j=0; j<n; j++)
            scanf("%d", &G[i][j]);
    printf("BFS Traversal: ");
    BFS(G, n, 0);
    return 0;
}

Algorithm
Step-by-step BFS procedure

The BFS algorithm proceeds systematically, expanding the frontier level by level. Each step is precisely defined to ensure all reachable vertices are visited exactly once.

1
Initialise Setup
Create a visited array of size V, initialised to 0. Create an empty queue. Choose the source vertex s.
2
Mark & Enqueue Source Start
Mark visited[s] = 1. Enqueue vertex s into the queue. Set front = rear = 0.
3
Check Queue Loop
While the queue is not empty (front ≤ rear), continue the traversal. If queue is empty, BFS is complete.
4
Dequeue Front Vertex Process
Remove the front vertex from the queue (increment front). Process it — print or record the vertex in the traversal order.
5
Scan All Adjacent Vertices Explore
For each vertex i from 0 to n-1, check if G[vertex][i] == 1 (there is an edge) and visited[i] == 0.
6
Enqueue Unvisited Neighbours Enqueue
For each unvisited adjacent vertex i, mark visited[i] = 1 and enqueue it. This prevents duplicates and ensures level-order processing.
7
Repeat Until Queue Empty Termination
Go back to Step 3. When the queue becomes empty, all reachable vertices have been visited. The BFS traversal is complete.
Why mark visited BEFORE enqueuing (not after dequeuing)?
If we mark visited only when dequeuing, the same vertex might be enqueued multiple times by different neighbours before it's processed — causing redundant work and potentially incorrect results. Marking it immediately upon enqueueing prevents this.

Procedure
Step-by-step instructions

Follow these steps to successfully compile, execute, and analyse the BFS program in C.

🛠️
Step 1 — Set Up Environment
Open a terminal and ensure GCC is installed (gcc --version). Create a new file bfs.c using a text editor or IDE. Copy the provided BFS program into the file.
⚙️
Step 2 — Compile the Program
Run: gcc bfs.c -o bfs. Resolve any compilation errors (check for missing semicolons, undeclared variables). A successful compile produces the executable bfs.
▶️
Step 3 — Execute the Program
Run: ./bfs (Linux/Mac) or bfs.exe (Windows). The program will prompt for the number of vertices and the adjacency matrix.
📥
Step 4 — Provide Input
Enter the number of vertices n (e.g., 5). Then enter the n×n adjacency matrix row by row. Use 1 if an edge exists between vertices i and j, else 0. The diagonal should be 0 (no self-loops).
👁️
Step 5 — Observe the Simulation
In the Simulation tab, use Generate Sample Graph or add nodes/edges manually. Press Start BFS and observe the queue grow and shrink as each vertex is processed level by level.
🔬
Step 6 — Analyse & Record
Note the BFS traversal order, the maximum queue size, the number of queue operations, and which vertices were at each level. Compare with DFS output if available.
📊
Step 7 — View Results
Navigate to the Result tab after the simulation completes to see the full traversal order, statistics, and complexity analysis auto-populated from your simulation run.
Tip: Use the Step Forward button in the simulation for granular manual control. The pseudocode panel on the left highlights the exact line being executed, and the queue panel on the right shows live queue state — making it easy to correlate each operation with the algorithm.
